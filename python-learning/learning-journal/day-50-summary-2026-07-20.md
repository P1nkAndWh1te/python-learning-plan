# Day 50 学习总结：API 文档与架构说明

今天没有新增业务功能，而是把 DocuAsk 当前能力整理成 API 文档和架构说明。

## 今天学到的重点

一个项目不只是“代码能跑”，还需要能讲清楚：

- 系统有哪些接口。
- 每个接口解决什么问题。
- 数据从哪里来，到哪里去。
- 每个 service 的边界是什么。
- 当前指标是什么。
- 当前限制是什么。

## 当前项目链路

现在 DocuAsk 可以这样理解：

```text
文档 -> 切分 -> embedding -> Chroma 持久化 -> 检索 -> 上下文 -> 来源引用 -> 评测
```

FastAPI 后端提供：

```text
/health
/documents
/qa
/evaluation
```

检索模式支持：

```text
vector
bm25
rrf
```

## 最重要的认知

文档不是装饰，而是项目工程化的一部分。

如果只说“我实现了 RAG 系统”，说服力不够。  
如果能说“我拆分了哪些模块，用哪些接口串联，用哪些指标评估，并记录了哪些限制”，项目可信度会高很多。

## 下一步

下一步可以考虑把 LLM 生成也接到 FastAPI 后端，让后端不仅返回 retrieved context，还能返回 final answer 和 sources。
