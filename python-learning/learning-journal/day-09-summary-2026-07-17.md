## 今日核心理解

### 1. Embedding 是什么

在今天的简化脚本里，embedding 是根据文本命中的关键词类别和数量，把文本转换成一个固定长度的数字向量。

例如：

```text
Concept order: [python, api, git, rag, data]
Vector:        [1,      0,   0,   0,   1]
```

含义是：

- `python` 维度命中 1 个关键词。
- `data` 维度命中 1 个关键词。
- 其他维度没有命中。

### 2. 余弦相似度看方向

余弦相似度比较的是向量方向，而不是数字大小完全相等。

例如：

```text
Query: [0, 0, 0, 3, 0]
Doc:   [0, 0, 0, 4, 0]
```

两个向量都只指向 `rag` 维度，所以方向完全一致，`Score = 1.0000`。

### 3. Score 为 1 的含义

`Score = 1.0000` 不代表关键词数量完全相同，而是代表两个向量在各个维度上的比例一致。

如果文档多出一个会改变方向的维度，分数就会下降。

### 4. 手写 embedding 和真实 embedding 的区别

今天的手写 embedding 只有 5 个维度，并且靠关键词命中生成，规则很简单。

真实 embedding 模型会把文本转换成几百或几千维向量，包含更丰富的语义、上下文和词语关系。即使两个文本没有完全相同的关键词，也可能因为意思接近而被判断为相似。

## 今日练习结果

### 1. API 查询

查询：

```text
怎么让 Python 程序调用接口获取数据？
```

结果：

```text
Rank 1: doc_api
Score: 0.9526
```

说明 API 文档和 query 的向量方向最接近。

### 2. RAG 查询

查询：

```text
RAG 怎么从知识库里找到相关文档？
```

结果：

```text
Rank 1: doc_rag
Score: 1.0000
```

说明 query 和 RAG 文档都只指向 `rag` 维度，方向完全一致。

### 3. 自己新增文档

新增：

```python
{
    "id": "doc_python_file",
    "text": "Python 可以读取文件，并把内容保存成 JSON 或 CSV。",
}
```

查询：

```text
Python 怎么读取文件并保存数据？
```

结果：

```text
Rank 1: doc_python_file
Score: 1.0000
```

说明新增文档被正确检索到。

### 4. Chroma 检索

Chroma 查询：

```text
怎么让 Python 程序调用接口获取数据？
```

结果：

```text
Rank 1
Distance: 1.0000
Metadata: {'source': 'doc_api'}
Document: Python 可以使用 requests 或 SDK 调用 API，获取远程服务的数据。

Rank 2
Distance: 5.0000
Metadata: {'source': 'doc_python_file'}
Document: Python 可以读取文件，并把内容保存成 JSON 或 CSV。
```

注意：Chroma 输出的是 `Distance`，不是我们手写脚本里的 `Score`。在这个输出里，距离越小越接近，所以 `doc_api` 排第一。

## 今日踩坑和注意事项

| 问题 | 原因 | 处理 |
|---|---|---|
| 终端中文乱码 | Windows 控制台编码显示问题 | 在脚本中设置 `sys.stdout.reconfigure(encoding="utf-8")` |
| Chroma 生成 `chroma_store/` | 本地向量数据库持久化文件 | 加入 `.gitignore`，不提交到 GitHub |
| Chroma 使用 `Distance` 而不是 `Score` | 向量数据库返回距离指标 | 距离越小越相似，和余弦分数越大越相似不同 |
